# 13. ROB

乱序处理器的核心矛盾是：**指令可以乱序执行，但结果必须按序提交**。ROB 就是解决这个矛盾的"秩序守护者"——它记录每条指令的状态，确保即使指令乱序完成，最终的效果也和顺序执行一模一样。

* ✅ 理解 ROB 分 Bank 设计的动机与实现

## 13.1 整体定位：ROB 是什么？

你可以把 ROB 想象成一座**档案柜**：

* 每条指令入队时在 ROB 中占一个位置（按程序顺序排列）
* 指令乱序执行、乱序写回，但写回时只更新 ROB 中的状态（标记为"已完成"）
* 只有 ROB 头部连续的"已完成"指令才能被**提交**——真正生效

ROB 的核心职责：**保证精确中断和按序提交**。没有 ROB，乱序处理器就无法在异常发生时恢复到一致的机器状态。

## 13.2 ROB Compress（ROB 压缩）

### 13.2.1 为什么需要压缩？

一条复杂指令（如向量指令、融合指令）可能在 Dispatch 阶段被拆分为**多个微操作**。如果每个 uop 都在 ROB 中占一项，ROB 的空间会很快耗尽，严重限制指令窗口大小。

ROB 压缩的核心思想：**一条指令的多个 uop 共享同一个 ROB 项**，用计数器跟踪完成进度。

### 13.2.2 uopNum 计数器

每个 ROB 项维护一个 <code>**uopNum**</code> 字段，记录**还有多少个 uop 未写回**：

| **状态** | **uopNum 值** | **含义** |
| --- | --- | --- |
| 刚入队 | 等于该指令的 uop 总数 | 所有 uop 都还没写回 |
| 部分 uop 写回 | 递减 | 剩余 uop 数 |
| 全部写回 | 0 | 该指令执行完毕，可以提交 |

```scala
def isWritebacked: Bool = !uopNum.orR
```

### 13.2.3 realDestSize 与 lastUop

压缩带来的一个问题是：多个 uop 可能写不同的物理寄存器，但只有**最后一个 uop 的写回**才是真正需要提交给架构态的。<code>**realDestSize**</code> 字段记录了真正有效的目标寄存器数量，<code>**lastUop**</code> 标记标识哪个是最后一个微操作。

| **字段** | **含义** | **比喻** |
| --- | --- | --- |
| <code>**uopNum**</code> | 剩余未写回的 uop 数 | 还有几个快递没到 |
| <code>**realDestSize**</code> | 真正有效的目标数 | 其中有几个是你要的 |
| <code>**lastUop**</code> | 最后一个微操作 | 最后一个快递到了才算齐 |

## 13.3 ROB Commit（ROB 提交）

### 13.3.1 提交条件

ROB 头部的指令可以被提交，需要满足以下条件：

| **条件** | **说明** |
| --- | --- |
| **valid** | 该 ROB 项有效（有指令占用） |
| **writebacked** | 所有 uop 已写回（uopNum = 0） |
| **无异常** | 该指令没有触发异常 |
| **前方无阻塞** | 没有未处理的异常或特殊指令阻塞提交 |

### 13.3.2 提交流程

```plain
ROB 头部检查
    │
    ├──→ 有异常？ ──→ 是 ──→ 暂停提交，触发异常处理
    │
    ├──→ 全部写回？ ──→ 否 ──→ 等待写回
    │
    └──→ 满足条件 ──→ 是 ──→ 每周期最多提交 CommitWidth 条指令
                              │
                              ├──→ 通知 Rab：更新架构寄存器映射
                              ├──→ 通知 LSQ：Store 可以真正写入内存
                              ├──→ 通知 CSR：fflags/vxsat 累加
                              └──→ 释放 ROB 项，推进 deqPtr
```

### 13.3.3 提交宽度

香山的 ROB 每周期可以提交 **CommitWidth** 条指令（通常为 6~8）。但实际提交数受限于：

* 头部连续的已写回项数量
* 是否遇到异常或特殊指令
* LSQ 的 Store 提交带宽

### 13.3.4 提交时的副作用

提交不只是"打个勾"，还会触发一系列**架构状态更新**：

| **副作用** | **目标** | **说明** |
| --- | --- | --- |
| **寄存器映射更新** | Rab → 架构 RAT | 确认物理寄存器归属 |
| **旧寄存器释放** | FreeList | 释放被覆盖的旧物理寄存器 |
| **Store 提交** | LSQ / SBuffer | Store 数据真正写入内存 |
| **fflags 累加** | CSR | 浮点异常标志累积 |
| **vxsat 累加** | CSR | 向量饱和标志累积 |
| **VTYPE 更新** | VTypeBuffer | 向量配置生效 |

## 13.4 ROB Exception（ROB 异常处理）

### 13.4.1 为什么异常必须按序处理？

乱序处理器中，指令可能乱序发现异常。但 RISC-V 规范要求：**只有程序顺序上第一条发生异常的指令才应该被处理**，后续指令的异常应该被忽略。

这就像排队看病——即使你比别人先查出问题，也必须按挂号顺序叫号。

### 13.4.2 异常检测来源

ROB 从多个写回端口收集异常信息：

| **来源** | **信号** | **异常类型** |
| --- | --- | --- |
| **执行单元写回** | <code>**exceptionWBs**</code> | 非法指令、断点、ECALL 等 |
| **Load/Store 写回** | <code>**exceptionWBs**</code> | 缺页、访问权限、地址不对齐 |
| **CSR 写回** | <code>**csrWBs**</code> | CSR 权限异常 |
| **分支/跳转写回** | <code>**redirectWBs**</code> | 误推测重定向 |

### 13.4.3 异常处理流程

```plain
① 执行单元检测到异常，写回 ROB
        │
        ▼
② ROB 标记该项的异常标志位
        │
        ▼
③ 等待该指令到达 ROB 头部（按序）
        │
        ▼
④ 确认是程序顺序上第一条异常指令？
    ├──→ 是 ──→ 暂停提交，触发异常陷入
    └──→ 否 ──→ 忽略（被更早的冲刷清除了）
        │
        ▼
⑤ 触发异常陷入：
    · 刷新 ROB 中该指令之后的所有项
    · 通知前端从异常处理程序取指
    · 更新 CSR（mepc、mcause、mtval 等）
```

### 13.4.4 中断安全

中断是特殊的"异常"——它不是由当前指令触发的，但在提交时需要检查。ROB 中每个项有一个 <code>**interrupt_safe**</code> 标志：

* 某些指令（如 CSR 指令、FENCE 指令）在完成前会修改处理器状态，**不能被打断**
* 只有 <code>**interrupt_safe**</code> 的指令之间才能响应中断
* 中断的优先级低于异常——如果同一条指令既有异常又有中断挂起，优先处理异常

## 13.5 Replay（重放）

### 13.5.1 什么是 Replay？

某些指令执行失败不是因为程序错误，而是因为**暂时性的资源冲突**——比如 TLB 缺失、Store 地址还未计算出来、MMIO 访问需要排队等。这些情况下，指令需要**重新发射**，而不是触发异常。

这就像快递送到了但家里没人——不是地址错了，只是暂时收不了，改天再送。

### 13.5.2 Replay 的触发场景

| **场景** | **原因** | **Replay 方式** |
| --- | --- | --- |
| **TLB 缺失** | 页表项不在 TLB 中，需要从内存加载 | 等 TLB 填充完成后重新发射 |
| **Store 地址未就绪** | Load 指令依赖的 Store 地址还没算出来 | 等 Store 地址就绪后重新发射 |
| **MMIO 访问** | 访问内存映射 IO，需要特殊处理 | 串行化处理 |
| **Bank 冲突** | 多条指令同时访问同一 RegCache Bank | 下一拍重新竞争 |

### 13.5.3 Replay 的实现

Replay 的核心是 <code>**replayInst**</code> 标志——当执行单元发现需要 Replay 时，在写回 ROB 时设置该标志。ROB 在提交时检测到 Replay，会：

1. **不提交该指令**
2. **冲刷流水线中该指令之后的所有指令**
3. **重新从该指令的 PC 开始取指和执行**

:::danger
🚨注意\
Replay 的代价很高——相当于一次小规模的流水线冲刷。因此处理器会尽量减少 Replay 的发生，比如通过 Store-to-Load Forwarding 避免地址未就绪的 Replay。

:::

## 13.6 ROB Bank（ROB 分体）

### 13.6.1 为什么 ROB 需要分 Bank？

ROB 的大小通常在 200 项以上，每项都有多个字段需要读写。尤其是在**提交阶段**，需要同时读取 ROB 头部连续 CommitWidth 项的信息——如果用单体 SRAM 实现，读端口数量将导致面积爆炸。

### 13.6.2 分 Bank 策略

香山将 ROB 分为 **8 个 Bank**，按 ROB 索引的低位交错分配：

```scala
Bank 0: ROB[0], ROB[8],  ROB[16], ...
Bank 1: ROB[1], ROB[9],  ROB[17], ...
Bank 2: ROB[2], ROB[10], ROB[18], ...
...
Bank 7: ROB[7], ROB[15], ROB[23], ...
```

### 13.6.3 分 Bank 的关键约束

CommitWidth 通常为 6~8，而 Bank 数为 8。这保证了一个有趣的设计约束：

***连续 CommitWidth 项一定分布在不同的 Bank 中。***

这意味着提交时**每个 Bank 最多被读一次**——没有 Bank 冲突，不需要仲裁，提交逻辑的时序显著改善。

### 13.6.4 分 Bank 的优势与代价

| **优势** | **代价** |
| --- | --- |
| 每个 Bank 读端口数减少 | 写回时需要路由到正确 Bank |
| 提交无 Bank 冲突 | 入队时需要计算目标 Bank |
| 单 Bank 面积减小 | Bank 间指针管理更复杂 |

## 13.7 ROB Timing Pressure（ROB 时序压力）

### 13.7.1 关键时序路径

| **路径** | **描述** | **严重程度** |
| --- | --- | --- |
| **写回 → 状态更新** | 多个写回端口同时更新 ROB 项 | 🔴 极高 |
| **提交 → 读取头部** | 读取连续 CommitWidth 项的信息 | 🔴 极高 |
| **异常检测 → 冲刷** | 发现异常后计算冲刷范围 | 🟡 中等 |
| **Rab 提交 → 架构 RAT 更新** | 提交时同步更新寄存器映射 | 🟡 中等 |

### 13.7.2 优化策略一：ROB 与 Rab 分离

ROB 负责指令的**生命周期管理**（有效、写回、异常），而 **Rab（Rename Archive Buffer）** 负责寄存器映射的**架构态更新**。两者解耦后：

* ROB 只需维护轻量的状态位，提交时通知 Rab 即可
* Rab 独立管理寄存器映射的提交与回滚，不与 ROB 的写回逻辑竞争时序

### 13.7.3 优化策略二：写回端口合并

ROB 需要接收所有执行单元的写回，端口数量可能多达 20+。香山通过**写回端口合并**减少实际写入端口数——同一周期写回同一 Bank 的请求会被合并或序列化。

### 13.7.4 优化策略三：Walk 路径优化

当发生冲刷（redirect）时，ROB 需要从当前指针**回退**到重定向点，逐项释放。这就是"Walk"过程。Walk 的速度直接影响恢复时间：

* 普通方式：每周期 Walk CommitWidth 项
* 优化方式：利用 Snapshot 快速恢复指针，跳过大量项

### 13.7.5 优化策略四：VTypeBuffer 独立管理

向量配置信息（VTYPE、VL）的提交逻辑与普通寄存器不同，香山将其独立为 **VTypeBuffer**，避免与 ROB 的主提交路径竞争时序。

## 13.8 ROB 的状态机

ROB 的运行可以分为三种状态：

| **状态** | **含义** | **触发条件** |
| --- | --- | --- |
| **s\_idle** | 正常运行：入队、写回、提交 | 默认状态 |
| **s\_walk** | 回退：冲刷后逐项释放 | 收到 redirect 信号 |
| **s\_special\_walk** | 特殊回退：向量异常等 | 向量 Load 异常触发 |

```plain
     ┌──────────────────────────┐
     │                          │
     ▼                          │
  s_idle ──→ (redirect) ──→ s_walk ──→ (walk完成) ──→ s_idle
     │                                              	▲
     └──→ (vecLoadExcp) ──→ s_special_walk ───────────┘
```

## 13.9 总结

## <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">✅</font><font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);"> 核心要点总结</font>

* **ROB Compress**：多 uop 共享同一 ROB 项，uopNum 计数器跟踪写回进度；realDestSize 标识有效目标；全部 uop 写回后才可提交
* **ROB Commit**：头部连续已写回项按序提交；提交触发架构 RAT 更新、旧寄存器释放、Store 提交、fflags/vxsat 累加；CommitWidth 受头部连续项数和 LSQ 带宽限制
* **ROB Exception**：异常必须按序处理——只有程序顺序上第一条异常指令触发陷入；interrupt\_safe 标志控制中断响应时机
* **Replay**：暂时性资源冲突导致指令重放（TLB 缺失、Store 地址未就绪、MMIO）；代价等于一次流水线冲刷
* **ROB Bank**：8 Bank 交错分配，保证提交无 Bank 冲突；每个 Bank 最多被读一次
* **ROB Timing Pressure**：关键路径在写回状态更新和提交头部读取；优化策略包括 ROB/Rab 分离、写回端口合并、Snapshot 加速 Walk、VTypeBuffer 独立

**核心原则**：ROB 的设计围绕\*\*"秩序与恢复"\*\*展开——秩序是指按序提交保证精确中断，恢复是指冲刷后快速回到一致状态。压缩和分 Bank 是在"秩序"约束下追求性能的工程手段，Walk 优化和 Rab 分离是在"恢复"约束下追求速度的关键技术。
