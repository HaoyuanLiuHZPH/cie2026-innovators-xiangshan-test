# 9. Execute Unit

在前面的章节里，我们看到了指令如何被派发、如何在发射队列中等待、如何通过旁路网络获取操作数。现在，终于到了指令真正"干活"的地方——执行单元。

:::info
⚡读完本章，你将能够：

* ✅ 理解香山后端八类执行单元的整体布局与分工
* ✅ 掌握每类执行单元的功能范围、延迟特征和写回目标
* ✅ 认识标量与向量执行单元的关键差异
* ✅ 了解控制流指令和 CSR 指令的特殊处理机制

:::

***

## 9.1 整体定位：执行单元是干什么的？

你可以把执行单元想象成一座**工厂的车间**——每个车间负责不同的工序：

* 有的车间做加法（ALU），又快又简单
* 有的车间做乘法，稍微慢一点
* 有的车间做除法，慢得令人发指
* 有的车间专门跑向量指令，一次能处理一大堆数据
* 还有的车间负责和"外部世界"打交道（访存、CSR）

每个执行单元由一个或多个**功能单元** 组合而成。同一个执行单元可以包含多个 Fu——就像一个车间里可以装好几台不同的机器，谁闲着谁上工。

***

## 9.2 八类执行单元总览

| **类别** | **功能单元** | **所属调度域** | **写回目标** | **典型延迟** |
| --- | --- | --- | --- | --- |
| **标量整数** | alu, mul, div, bku, i2f, i2v | Int | 整数 RF / 浮点 RF | 0~数十拍 |
| **标量浮点** | falu, fmac, fcvt, fDivSqrt, fcmp, f2v | FP | 浮点 RF | 2~数十拍 |
| **标量访存** | ldu, stu, mou | Int/Mem | 整数 RF | 3~数十拍 |
| **向量整数** | vipu, vialuF, vppu, vimac, vidiv | VF | 向量 RF / V0 RF | 1~数十拍 |
| **向量浮点** | vfalu, vfma, vfdiv, vfcvt | VF | 向量 RF | 1~数十拍 |
| **向量访存** | vldu, vstu, vsegldu, vsegstu | VF/Mem | 向量 RF | 3~数十拍 |
| **控制流** | jmp, brh | Int | 整数 RF + 重定向 | 1~2 拍 |
| **CSR** | csr, fence | Int | 整数 RF + 重定向 | 0~数拍 |

:::warning
❤️新手建议\
你不需要一次记住所有功能单元的名字。核心只需理解：**标量管单条数据，向量管一捆数据；整数/浮点/访存是三大计算类型，控制流和 CSR 是特殊指令**。

:::

***

## 9.3 Scalar Integer（标量整数）

标量整数执行单元是处理器中**最忙碌的车间**——几乎所有程序都在不停地做加、减、与、或、移位。它们的特点是：**快、多、简单**。

### 9.3.1 功能单元一览

| **功能单元** | **职责** | **延迟** | **比喻** |
| --- | --- | --- | --- |
| **ALU** | 算术逻辑运算：加、减、与、或、异或、移位 | 0 拍 | 快餐——随点随到 |
| **MUL** | 乘法运算 | 1 拍 | 盖浇饭——稍等就好 |
| **DIV** | 除法运算 | 数十拍（迭代） | 炖菜——急不得 |
| **BKU** | 位操作：CLZ、CTZ、CPOP、ROLC 等 | 0~1 拍 | 小菜——快速上桌 |
| **I2F** | 整数转浮点 | 2 拍 | 跨车间调货 |
| **I2V** | 整数送入向量单元 | 1 拍 | 跨车间调货 |

### 9.3.2 关键特征

* **ALU 是 0 延迟的**——发射当拍就有结果，可以通过 Forward 通路当拍前递
* **DIV 是阻塞式的**——执行期间占据功能单元，其他指令只能等待，因此除法器是流水线中最大的瓶颈之一
* **I2F 跨域写回**：结果写浮点寄存器堆，需要走浮点写回端口，Bypass Network 会自动处理延迟对齐

***

## 9.4 Scalar Floating-point（标量浮点）

浮点运算比整数运算复杂得多——需要对齐阶码、做尾数运算、规格化结果。香山的浮点执行单元采用了**多级流水**设计。

### 9.4.1 功能单元一览

| **功能单元** | **职责** | **延迟** | **写回** |
| --- | --- | --- | --- |
| **FALU** | 浮点加减、浮点比较相关辅助运算 | 2 拍 | 浮点 RF |
| **FMAC** | 浮点乘加融合（FMA） | 3 拍 | 浮点 RF |
| **FCVT** | 浮点格式转换：单精度↔双精度 | 2 拍 | 浮点 RF |
| **FDivSqrt** | 浮点除法和开方 | 数十拍 | 浮点 RF |
| **FCMP** | 浮点比较 | 1 拍 | 整数 RF（结果写回整数寄存器） |
| **F2V** | 浮点数据送入向量单元 | 1 拍 | 向量 RF |

### 9.4.2 关键特征

* **FMAC 是浮点运算的核心**——绝大多数浮点指令（乘、加、乘加）都走 FMAC，延迟 3 拍
* **FDivSqrt 是最慢的浮点运算**——采用 SRT 算法迭代，延迟不固定，且执行期间占用功能单元
* **FCMP 的结果写回整数寄存器堆**——这就是 F2I（Float to Int）旁路的来源，Bypass Network 需要处理跨域延迟
* **所有浮点运算都需要 FRM（舍入模式）**——从 CSR 中读取，作为额外输入

***

## 9.5 Scalar Memory（标量访存）

访存指令是处理器与内存之间的桥梁。它们不是"算数"的，而是"搬运"的——把数据从内存搬到寄存器，或者反过来。

### 9.5.1 功能单元一览

| **功能单元** | **职责** | **延迟** | **写回** |
| --- | --- | --- | --- |
| **LDU** | 标量 Load：从内存读取数据 | 3 拍起（Cache 命中） | 整数 RF / 浮点 RF |
| **STU** | 标量 Store：将数据写入内存 | 3 拍起 | 无（Store 不写回） |
| **MOU** | 原子内存操作（AMO） | 数拍 | 整数 RF |

### 9.5.2 关键特征

* **Load 延迟不确定**——Cache 命中 3 拍，Cache 缺失可能几十拍甚至上百拍。这是整个处理器中延迟最不可预测的操作
* **Store 不写回寄存器堆**——它只把地址和数据送到 Store Queue，等 ROB 提交后才真正写入内存
* **Load 需要 PC**——用于预测错误的回滚和调试
* **Load 可能被取消**——当发现地址违约定向时，Load 的结果无效，需要取消下游指令的唤醒。这就是 Issue Queue 中 <code>**ldCancel**</code> 的来源

:::warning
💡核心思想\
访存指令的特殊性在于**延迟不确定**。Cache 命中时很快，Cache 缺失时极慢。这种不确定性会向后传播——依赖 Load 结果的指令必须等 Load 确认无误后才能放心使用其结果。这就是 Load Wakeup + Cancel 机制存在的根本原因。

:::

***

## 9.6 Vector Integer（向量整数）

向量执行单元是香山中最复杂的车间——一条向量指令可以同时处理多个数据元素，就像一条流水线同时生产多个产品。

### 9.6.1 功能单元一览

| **功能单元** | **职责** | **延迟** | **写回** |
| --- | --- | --- | --- |
| **VIPU** | 向量整数逐元素运算：加减、逻辑、移位、比较 | 1 拍 | 向量 RF |
| **VIALUF** | 向量整数 ALU Fix：定点运算、宽位运算 | 1 拍 | 向量 RF |
| **VPPU** | 向量置换：VPERM、VSLIDE、VGATHER 等 | 1 拍 | 向量 RF |
| **VIMAC** | 向量整数乘加 | 3 拍 | 向量 RF |
| **VIDIV** | 向量整数除法 | 数十拍 | 向量 RF |

### 9.6.2 关键特征

* **向量指令可能写 V0 寄存器**——V0 是向量掩码寄存器，部分指令会更新它，需要专门的 V0 写回端口
* **向量指令可能写 VL 寄存器**——VSET 类指令会更新 VL（向量长度），VL 有独立的写回通道
* **需要 VXRM**——向量定点舍入模式，从 CSR 中读取
* **需要 VTYPE**——向量配置信息（SEW、LMUL 等），VSET 指令会写入
* **一条向量指令可能拆分为多个微操作**——尤其是 segment 指令和长向量指令

***

## 9.7 Vector Floating-point（向量浮点）

向量浮点执行单元是标量浮点的"向量版"——一条指令同时处理多个浮点元素。

### 9.7.1 功能单元一览

| **功能单元** | **职责** | **延迟** | **写回** |
| --- | --- | --- | --- |
| **VFALU** | 向量浮点加减、比较 | 1 拍 | 向量 RF |
| **VFMA** | 向量浮点乘加 | 3 拍 | 向量 RF |
| **VFDIV** | 向量浮点除法和开方 | 数十拍 | 向量 RF |
| **VFCVT** | 向量浮点格式转换 | 1 拍 | 向量 RF |

### 9.7.2 关键特征

* **与标量浮点共享 FRM**——舍入模式从同一个 CSR 读取
* **VFMA 是向量浮点的核心**——延迟 3 拍，是向量浮点运算的主力
* **VFDIV 是最慢的向量运算**——每拍迭代一个元素，元素数 × 单元素延迟
* **需要额外的 OG2 流水级**——向量执行单元比标量单元多一级流水，因此旁路需要 Bypass2 延迟

:::warning
❤️新手建议\
向量浮点与向量整数的最大区别在于**写回目标**——向量浮点只写向量 RF，不会写 V0/VL；而向量整数中的某些指令（如 VSET）会写 VL 和 VTYPE。这是区分它们的关键。

:::

***

## 9.8 Vector Memory（向量访存）

向量访存是所有执行单元中**最复杂的**——一条向量 Load/Store 指令可能涉及数十个元素、多个地址计算、Segment 拆分，甚至不确定的执行时间。

### 9.8.1 功能单元一览

| **功能单元** | **职责** | **延迟** | **写回** |
| --- | --- | --- | --- |
| **VLDU** | 向量 Load | 3 拍起（Cache 命中） | 向量 RF |
| **VSTU** | 向量 Store | 3 拍起 | 无 |
| **VSEGLDU** | 向量 Segment Load | 3 拍起 | 向量 RF |
| **VSEGSTU** | 向量 Segment Store | 3 拍起 | 无 |

### 9.8.2 关键特征

* **一条向量访存指令可能被拆成多个 Flow**——Dispatch 阶段的 <code>**conserveFlows**</code> 机制就是为它服务的
* **Unit-stride 指令的 Flow 数不确定**——只有地址计算完成后才知道是否需要拆分，因此 Dispatch 阶段保守估计为 2
* **Segment 指令只能从端口 0 发射**——因为它们需要的资源最多，不能和其他向量访存指令并行入队
* **VLEFF（Vector Load Effective）指令**——一种特殊的 unit-stride Load，实际加载的元素数取决于 VL，在 Dispatch 阶段用 <code>**isfofFixVlUop**</code> 标记

***

## 9.9 Control Flow（控制流：分支与跳转）

控制流指令不"计算"数据，它们决定**程序的走向**——下一条该执行哪条指令？它们的特殊性在于：可能触发**流水线冲刷**。

### 9.9.1 功能单元一览

| **功能单元** | **职责** | **延迟** | **特殊输出** |
| --- | --- | --- | --- |
| **JMP** | 跳转：JAL、JALR、AUIPC | 1 拍 | 重定向（目标地址） |
| **BRH** | 分支：BEQ、BNE、BLT 等 | 1~2 拍 | 重定向（跳转/不跳转） |

### 9.9.2 关键特征

* **可能产生重定向**——如果分支预测错误，需要冲刷流水线并从正确路径重新取指。这是处理器性能最大的杀手之一
* **需要 PC 和 FTQ 指针**——用于计算跳转目标和比对预测结果
* **需要预测信息**——分支预测器给出的预测方向和目标，用于与实际结果比对
* **JMP 指令会操作 RAS**——JAL 是 Push，JALR 可能是 Pop，用于函数调用/返回栈的维护
* **JMP 和 BRH 会设置 blockBackward**——确保后续指令不会在控制流确定之前提前发射

:::warning
💡核心思想\
控制流指令的核心矛盾是：**预测可能对，也可能错**。对了就万事大吉，错了就要回滚。香山的设计哲学是"大胆预测，小心验证"——先按预测往下跑，等实际执行结果出来再确认。预测正确的开销为 0，预测错误的开销是十几拍的流水线冲刷。

:::

***

## 9.10 Control and Status Register（CSR 与控制寄存器）

CSR 指令是处理器中最"特殊"的一类——它们不直接参与计算，而是**读写处理器的内部控制状态**。它们就像是工厂的"管理办公室"——发号施令、查看报表、修改参数。

### 9.10.1 功能单元一览

| **功能单元** | **职责** | **延迟** | **特殊输出** |
| --- | --- | --- | --- |
| **CSR** | 读写 CSR 寄存器：mstatus、mepc、satp 等 | 0~数拍 | 重定向 + 异常 |
| **FENCE** | 内存屏障：FENCE、FENCE.I | 0 拍（但会阻塞流水线） | 无 |

### 9.10.2 关键特征

* **CSR 指令可能产生重定向**——比如写 <code>**satp**</code> 切换页表后需要冲刷 TLB 和流水线；写 <code>**mepc**</code> 后执行 MRET 也会重定向
* **CSR 指令可能产生异常**——权限不足、非法 CSR 编号等
* **FENCE 指令设置 blockBackward**——确保内存屏障之前的所有访存指令都完成后，才允许后续指令继续
* **CSR 需要触发器**——用于在 CSR 读写时触发调试模式
* **CSR 实现极其复杂**——香山的 CSR 子系统包含数十个模块，覆盖机器模式、监督模式、虚拟化、调试、中断等，是代码量最大的功能单元之一

### 9.10.3 CSR 的分类

香山的 CSR 子系统按照 RISC-V 特权架构分层组织：

| **层级** | **对应模块** | **覆盖的 CSR** |
| --- | --- | --- |
| **Machine Level** | MachineLevel.scala | mstatus、mepc、mcause、mtvec... |
| **Supervisor Level** | SupervisorLevel.scala | sstatus、sepc、scause、stval... |
| **Virtual Supervisor** | VirtualSupervisorLevel.scala | vsstatus、vsepc... |
| **Hypervisor** | HypervisorLevel.scala | hstatus、hgeip... |
| **Debug** | Debug.scala | dcsr、dpc、dscratch... |
| **PMA/PMP** | PMAEntryModule / PMPEntryModule | 物理内存属性 / 物理内存保护 |

## 9.11 执行单元的延迟与唤醒关系

理解执行单元的延迟，是理解整个后端唤醒机制的基础。下图总结了关键延迟：

```plain
发射 ──→ OG0 ──→ OG1 ──→ OG2 ──→ 执行 ──→ 写回
         (选通)  (读RF)  (向量专用)          (唤醒)

ALU    : [0拍] ─────────────────────── 立即出结果 → Forward
MUL    : [1拍] ─────────────────────── 下一拍出结果 → Bypass
FMAC   : [3拍] ─────────────────────── 3拍后出结果 → IQ Wakeup
FDivSqrt: [数十拍] ──────────────────── 不确定延迟 → IQ Wakeup + Uncertain
LDU    : [3拍+] ────────────────────── Cache命中3拍，缺失更长 → IQ Wakeup + ldCancel
VFMA   : [3拍+OG2] ────────────────── 多一级OG2流水 → Bypass2
```

### 唤醒类型对照

| **唤醒类型** | **适用场景** | **特点** |
| --- | --- | --- |
| **Forward** | 0 拍延迟的 ALU | 当拍前递，最快 |
| **Bypass** | 1~2 拍延迟的 MUL/FALU | 1 拍后前递 |
| **Bypass2** | 带OG2的向量单元 | 2 拍后前递，跨域 |
| **IQ Wakeup** | 多拍延迟的 FMAC/VFMA | 通过 MultiWakeupQueue 延迟唤醒 |
| **Uncertain Wakeup** | 不确定延迟的 FDivSqrt/LDU | 可能被取消 |

## 9.12 总结

## <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">✅</font><font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);"> 核心要点总结</font>

* **标量整数**：ALU（0拍）、MUL（1拍）、DIV（数十拍）；ALU 走 Forward，是后端最快的路径
* **标量浮点**：FMAC（3拍）是主力，FDivSqrt 最慢；FCMP 结果写回整数 RF，是 F2I 旁路的来源
* **标量访存**：Load 延迟不确定，Store 不写回；Load Cancel 是后端唤醒取消机制的核心驱动
* **向量整数/浮点**：一条指令处理多个元素，可能写 V0/VL；需要额外的 OG2 流水级，走 Bypass2 旁路
* **向量访存**：最复杂的执行单元，一条指令可能拆成多个 Flow；Segment 指令限端口 0
* **控制流**：JMP/BRH 可能触发重定向，预测错误代价极高；设置 blockBackward 阻止后续指令抢先
* **CSR**：读写处理器内部状态，可能产生重定向和异常；FENCE 强制内存顺序；实现涵盖全部特权层级

**核心原则**：执行单元的多样性决定了后端数据通路的复杂性——不同的延迟需要不同的唤醒机制，不同的写回目标需要不同的旁路通道，不同的特殊行为需要不同的流水线控制。理解执行单元的特征，就是理解整个后端设计的关键。
